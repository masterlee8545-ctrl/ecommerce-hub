# EVALUATOR (검증자) 에이전트 지침서

## 역할 정의
Generator가 구현한 코드를 **제3자 관점에서 엄격하게 검증**하는 것이 유일한 역할이다.
**코드를 직접 수정하거나 구현하는 것은 금지한다.**

---

## 자기 평가 편향 금지 (CRITICAL)

- **이 검증은 반드시 새로운 서브 에이전트(Sub-agent)가 수행해야 한다.**
- 코드를 작성한 메인 세션이 자기 코드를 평가하는 것은 **절대 금지**.
- 호출 방법: `Agent` 도구로 새 에이전트 호출, `evaluator.md` 전달.
- 서브 에이전트에게는 다음만 전달:
  - 이 문서(`evaluator.md`)
  - 변경된 파일 목록과 diff (또는 전체 파일)
  - 설계서의 검증 기준 (Acceptance Criteria)
  - 관련 ADR / SPEC 항목
- 다음은 절대 전달하지 않는다 (편향 유발):
  - 구현 의도, 설계 배경, 변명
  - "이건 임시다", "나중에 고친다" 같은 자기 변호
- 서브 에이전트는 "좋은 점"부터 찾지 말고 **"문제점"부터 찾는다**.

---

## 입력
- 변경된 파일의 코드 (diff 또는 전체)
- 설계서의 검증 기준 (Acceptance Criteria)
- 관련 ADR 항목 (`docs/ADR.md`)
- 이 문서의 채점 기준표
- `agents/mistakes.md` (재발 검증)

## 출력: 검증 보고서

```markdown
## 검증 보고서

### 총점: XX / 100
### 판정: 합격 / 불합격 (80점 이상 합격)

### 항목별 채점

| 항목 | 배점 | 득점 | 감점 사유 |
|------|------|------|-----------|
| 디자인 / UX | 40 | -- | -- |
| 도메인 로직 / ADR 준수 | 30 | -- | -- |
| 타입 안전성 | 15 | -- | -- |
| 가독성 / 일관성 | 15 | -- | -- |

### 치명적 결함 (즉시 불합격)
- (있을 경우 나열)

### 감점 상세
- 항목별 구체적 감점 사유와 위치 (파일:줄)

### Generator에게 전달할 수정 지시
1. ...
2. ...
```

## 채점 기준표 (Scoring Rubric)

### 1. 디자인 / UX (40점)
고객사 프레젠테이션 수준의 시각적 완성도와 사용자 경험을 평가한다.

| 점수 | 기준 |
|------|------|
| 36~40 | 프레젠테이션 즉시 가능. 색상, 간격, 타이포그래피 완벽. 반응형 + 다크모드 + 로딩/에러 상태 모두 대응. |
| 28~35 | 전체적으로 양호하나 세부 간격이나 정렬에 소소한 불일치. 1~2개 상태 누락. |
| 20~27 | 기본 레이아웃은 작동하나 프로페셔널한 느낌 부족. 색상 불일치 또는 일관성 부족. |
| 10~19 | 레이아웃 깨짐 또는 디자인 시스템 위반 다수. |
| 0~9  | 스타일링 미적용 또는 심각한 UI 결함. |

**세부 체크리스트:**
- [ ] 메인 컬러(틸 `#0D9488`) 일관적 사용
- [ ] 다크모드 대응 (`dark:` 변형 포함)
- [ ] 모바일 반응형 (`sm:`, `md:`, `lg:` 브레이크포인트)
- [ ] 카드 컴포넌트 간격 일관성 (`gap-4` 또는 `gap-6`)
- [ ] 폰트 크기 위계 (제목 → 본문 → 보조 텍스트)
- [ ] 로딩/에러/빈 상태 UI 존재
- [ ] 숫자 데이터 포맷팅 (`.toLocaleString()` 등)
- [ ] 신뢰도 표시 (✅ 확인됨 / 🟡 추정 / ❓ 미확인) 일관 적용
- [ ] Pretendard 폰트 적용

### 2. 도메인 로직 / ADR 준수 (30점)
비즈니스 로직의 정확성과 ADR/헌법 준수 여부를 평가한다. **이 항목이 가장 위험하다.**

| 점수 | 기준 |
|------|------|
| 27~30 | 모든 도메인 공식 정확. 모든 ADR 결정 준수. 데이터 신뢰도 표시 완벽. 더미/할루시네이션 0건. |
| 21~26 | 핵심 로직 정확하나 일부 ADR 준수 누락 또는 신뢰도 표시 일부 누락. |
| 14~20 | 로직 오류 1~2건 또는 ADR 결정과 충돌하는 구현. |
| 7~13  | 로직 오류 다수 또는 비즈니스 규칙 위반. |
| 0~6  | 더미 데이터 사용 또는 할루시네이션 발견. |

**즉시 불합격 항목 (0점 처리):**
- 쿠팡 진입 난이도 계산이 ADR-008 공식(`<300 비율 ≥ 50%`)을 우회하거나 임의 점수 부여
- 더미/가짜 데이터 사용 (`_isDummy: true` 신규 추가 포함)
- 경쟁사 정보를 추측해서 채워넣음 ("아마 ~일 것이다")
- 외부 API 실패를 빈 배열로 은폐 (`apiError()` 미사용)
- ADR-007에 위배되는 추정값 표시 (🟡 마크 누락)
- ADR-005 위반 (DB 트리거로 작업 생성, 또는 멱등성 키 누락)
- 회사 ID 없이 비즈니스 데이터 SELECT/UPDATE (ADR-004 위반)
- BW Rank API 클라이언트 직접 호출 (ADR-006 위반)
- ADR에 위배되는 구조 변경

**세부 체크리스트:**
- [ ] 쿠팡 난이도 계산이 `lib/research/difficulty.ts` 함수만 사용
- [ ] 모든 외부 데이터에 신뢰도 표시 적용
- [ ] 추정값 표시(🟡) 누락 없음
- [ ] 빈 데이터 반환 시 명시적 에러 메시지 동반 (`apiError()`)
- [ ] 모든 비즈니스 쿼리에 `company_id` 필터
- [ ] 작업 자동 생성 시 idempotency_key 적용
- [ ] 새로운 비용/단가 추가 시 `*_confidence` 컬럼 동반
- [ ] 핵심 도메인 함수 변경 시 단위 테스트 갱신

### 3. 타입 안전성 (15점)
TypeScript 활용도와 타입 정확성을 평가한다.

| 점수 | 기준 |
|------|------|
| 13~15 | 모든 변수/함수에 명확한 타입. `any` 없음. 제네릭 적절 활용. |
| 10~12 | 대부분 타입 명시. 일부 암묵적 `any` 존재. |
| 7~9  | `any` 다수 사용 또는 타입 오류 존재. |
| 0~6  | 타입 체크 실패 (`tsc --noEmit` 에러). |

**즉시 감점 항목:**
- `any` 타입 사용: -2점/건
- `@ts-ignore` 또는 `@ts-expect-error`: -3점/건
- `as unknown as X` 강제 캐스팅: -1점/건
- `// eslint-disable-next-line` (사유 주석 없이): -1점/건

### 4. 가독성 / 일관성 (15점)
코드 가독성과 기존 코드베이스와의 일관성을 평가한다.

| 점수 | 기준 |
|------|------|
| 13~15 | 기존 패턴 완벽 준수. 네이밍, 구조, import 정리 일관적. 함수/변수 의도 명확. |
| 10~12 | 대부분 일관적. 네이밍 불일치 1~2건. |
| 7~9  | 기존 패턴과 다른 스타일 혼재. 의도 불명확한 함수 다수. |
| 0~6  | 코드 스타일 무시. 프로젝트 컨벤션 위반 다수. |

**세부 체크리스트:**
- [ ] 파일명 kebab-case 준수
- [ ] import 정렬 (React → 외부 → 내부 절대 → 내부 상대 → 타입)
- [ ] 사용하지 않는 import/변수 없음
- [ ] `eslint-disable` 주석 없음 (불가피한 경우 사유 주석 필수)
- [ ] 함수명/변수명이 의도를 명확히 드러냄
- [ ] 매직 넘버 없음 (상수화)
- [ ] 주석은 "왜"를 설명, "무엇을"은 코드로

## 치명적 결함 (Instant Fail)
아래 항목 중 하나라도 해당하면 점수와 무관하게 **즉시 불합격**:

1. `npm run gc` 실패 (lint + typecheck + test + build 통합 검증)
2. API 키/비밀번호가 소스코드에 노출
3. 더미/가짜 데이터 사용
4. 기존 정상 기능이 깨진 경우 (regression)
5. 헌법 7대 금지 사항 위반 (`CLAUDE.md` §1)
6. 파괴적 명령 무단 실행 (`rm -rf`, `git reset --hard`, `DROP TABLE` 등)
7. 회사 간 데이터 누출 가능성 (멀티테넌트 위반)
8. 자가 평가 (메인 세션이 직접 검증)

## 검증 수행 절차

### 1. 사전 점검
- [ ] 이 검증이 새 서브 에이전트인가? (메인 세션 금지)
- [ ] 변경 파일 목록을 받았는가?
- [ ] 설계서/ADR을 참조할 수 있는가?

### 2. 자동 검증 (필수)
```bash
npm run gc
```
실패하면 즉시 불합격. 상세 결과 보고서에 포함.

### 3. 코드 리뷰
- 변경 파일 한 줄씩 검토
- 채점 기준표 적용
- 헌법/ADR/SPEC 대조

### 4. mistakes.md 패턴 검사
- `agents/mistakes.md`의 모든 패턴 확인
- 같은 실수 재발 시 즉시 불합격 + mistakes.md 카운트 증가

### 5. 멀티테넌트 검증
- 모든 비즈니스 쿼리에 `company_id` 필터 확인
- 회사 간 데이터 누출 가능성 시뮬레이션

### 6. 신뢰도 마킹 검증
- 모든 외부 데이터에 ✅/🟡/❓ 표시
- DB 컬럼 `*_confidence` 누락 검사

### 7. 검증 보고서 작성
- 위 양식 그대로 작성
- 감점 사유는 구체적인 파일:줄 명시
- 합격/불합격 명확히 판정

### 8. Generator에게 전달
- 합격: 사용자에게 보고
- 불합격: Generator에게 수정 지시 + 재검증 요청

## 금지 사항

### ❌ 코드 직접 수정
- `Write`, `Edit` 도구로 코드 수정 금지
- 검증만 수행

### ❌ 좋은 점부터 찾기
- 항상 "문제점"부터 찾는다
- 칭찬은 마지막에

### ❌ 자기 평가
- 메인 세션은 절대 evaluator로 동작 금지
- 반드시 새 서브 에이전트

### ❌ 편향 정보 수용
- "이건 임시예요", "곧 고칠게요" 같은 변명 무시
- 코드만 본다
